# 开发规范

本文档定义 EasyBX 项目的代码风格、命名约定、Git 工作流和 AI 协作协议，确保代码库长期可维护。

---

## 1. 语言与运行时要求

| 维度 | 规格 |
|------|------|
| **前端语言** | TypeScript 5.x |
| **后端语言** | Python 3.11+ |
| **前端运行时** | Node.js >= 18.17 |
| **后端运行时** | Python 3.11+ |
| **前端包管理器** | npm >= 9.0（禁止使用 yarn / pnpm） |
| **后端包管理器** | pip + requirements.txt |
| **TypeScript 严格模式** | `strict: true`，禁止 `any`（特殊情况须 `// eslint-disable-next-line @typescript-eslint/no-explicit-any` 并注释理由） |
| **Python 类型注解** | 所有函数必须标注类型注解，使用 `from __future__ import annotations` |

| **测试文件（后端）** | `test_*.py` | `test_ocr_service.py` |

### 3.2 代码命名

| 元素 | 命名风格 | 示例 |
|------|---------|------|
| **React 组件** | `PascalCase` | `InvoiceCard` |
| **函数（TS）** | `camelCase` | `formatDate` |

## 4. Python 特定规范

### 4.1 类型注解

所有函数参数和返回值必须标注类型。

```python
from __future__ import annotations

# ✅ 正确
def process_invoice(file_path: str, user_id: int) -> dict[str, Any]:
    ...

# ❌ 错误 — 缺少类型注解
def process_invoice(file_path, user_id):
    ...
```

### 4.2 导入排序

按以下顺序分组，组间空一行：

```python
# 1. 标准库
from __future__ import annotations
import os
from datetime import datetime

# 2. 第三方库
from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session

# 3. 项目内部模块
from app.models.invoice import Invoice
from app.services.ocr_service import OcrService
```

### 4.3 ORM 模型定义规范

```python
# ✅ 正确 — SQLAlchemy 2.0 风格
from sqlalchemy import String, Integer, Float, Boolean
from sqlalchemy.orm import Mapped, mapped_column
from app.database import Base

class Invoice(Base):
    __tablename__ = "invoices"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    invoice_no: Mapped[str] = mapped_column(String(50))
    amount: Mapped[float] = mapped_column(Float)
    is_substitute: Mapped[bool] = mapped_column(Boolean, default=False)
```

### 4.4 Service 层模式

```python
# ✅ 正确 — Service 类使用类方法，不依赖实例状态
class OcrService:
    @staticmethod
    def recognize(file_path: str) -> OcrResult:
        ...

# ❌ 错误 — 不必要的实例化
class OcrService:
    def recognize(self, file_path: str) -> OcrResult:
        ...
```

### 4.5 FastAPI 路由规范

```python
# ✅ 正确 — 路由只做参数校验和调用 service
from fastapi import APIRouter, Depends

router = APIRouter(prefix="/api/invoices", tags=["invoices"])

@router.post("/")
async def create_invoice(
    file: UploadFile = File(...),
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user),
):
    result = await invoice_service.create(db, file, current_user.id)
    return JSONResponse(content=result, status_code=201)
```

### 4.6 避免的写法

```python
# ❌ 避免：在路由中直接写业务逻辑
@router.post("/")
async def create_invoice(file: UploadFile = File(...)):
    # 直接操作数据库、调用 OCR 等
    ...

# ✅ 替代方案：路由只负责 HTTP，逻辑放在 service
@router.post("/")
async def create_invoice(file: UploadFile = File(...)):
    result = await invoice_service.create(file)
    return result
```

```python
# ❌ 避免：全局可变变量
invoice_cache = []

# ✅ 替代方案：明确依赖注入
def get_cached_invoices(db: Session) -> list[Invoice]:
    ...
```

```python
# ❌ 避免：魔法数字
if invoice.amount > 5000:
    ...

# ✅ 替代方案：命名常量
MAX_AMOUNT_NO_APPROVAL = 5000
if invoice.amount > MAX_AMOUNT_NO_APPROVAL:
    ...
```

## 5. React 特定规范

### 5.1 组件设计原则

- **单一职责**：每个组件只做一件事。如果组件需要滚动才能看完其职责描述，说明它太大了。
- **原子化组合**：`ui/` 目录存放基础组件（Button、Input、Modal），业务组件放在 `invoices/`、`batches/`、`exports/` 目录下组合使用。
- **数据流向**：数据从 store（Zustand）→ 页面组件 → 业务组件 → UI 组件，禁止反向传递。

### 5.2 组件文件结构

```tsx
// web/src/components/invoices/InvoiceCard.tsx

// 1. 类型导入
import type { Invoice } from '@/types/invoice';

// 2. 依赖导入
import { useInvoiceStore } from '@/stores/invoiceStore';
import { formatCurrency } from '@/lib/utils';

// 3. 组件 Props 类型定义
interface InvoiceCardProps {
  invoice: Invoice;
  onEdit?: (id: number) => void;
}

// 4. 组件定义
export function InvoiceCard({ invoice, onEdit }: InvoiceCardProps) {
  return (
    <div>
      <span>{invoice.invoice_no}</span>
      <span>{formatCurrency(invoice.amount)}</span>
    </div>
  );
}
```

### 5.3 导出规则

- **始终使用命名导出**（`export function` / `export const`），禁止默认导出
- 例外：`React.lazy()` 动态加载时允许默认导出

```tsx
// ✅ 正确
export function InvoiceCard({ invoice }: Props) { ... }

// ❌ 错误
export default function InvoiceCard({ invoice }: Props) { ... }
```

### 5.4 导入排序

```tsx
// 按以下顺序分组，组间空一行：
// 1. React / 框架
import { useState, useCallback } from 'react';
import { Link } from 'react-router-dom';

// 2. 第三方库
import { useForm } from 'react-hook-form';
import { Upload } from 'lucide-react';

// 3. 项目内部 - API / Store / Hooks
import { useInvoiceStore } from '@/stores/invoiceStore';
import { useInvoices } from '@/hooks/useInvoices';

// 4. 项目内部 - 组件
import { Button } from '@/components/ui/Button';
import { InvoiceCard } from '@/components/invoices/InvoiceCard';

// 5. 项目内部 - 类型 / 工具
import type { Invoice } from '@/types/invoice';
import { formatDate } from '@/lib/utils';

// 6. 样式文件
import './styles.css';
```

### 5.5 Hooks 使用规范

- **命名**：以 `use` 开头，`camelCase`，描述功能而非实现
- **存放位置**：业务 hooks 放在 `web/src/hooks/`，组件内部逻辑不必抽取 hooks
- **禁止条件调用**：严格遵守 React Hooks 规则，不在循环/条件/嵌套函数中调用 hooks

```tsx
// ✅ 正确
export function useInvoices() {
  const { invoices, fetchInvoices } = useInvoiceStore();

  useEffect(() => {
    fetchInvoices();
  }, [fetchInvoices]);

  return { invoices };
}
```

## 6. 样式规范

### 6.1 唯一样式方案：Tailwind CSS

- 所有样式使用 Tailwind 原子类，禁止写自定义 CSS 文件
- 动态类名使用 `clsx` 或 `tailwind-merge` 组合

```tsx
// ✅ 正确
import { cn } from '@/lib/utils';

export function Button({ variant = 'primary', className }: Props) {
  return (
    <button
      className={cn(
        'rounded-md px-4 py-2 text-sm font-medium',
        variant === 'primary' && 'bg-blue-600 text-white',
        variant === 'ghost' && 'bg-transparent text-gray-600',
        className,
      )}
    />
  );
}
```

### 6.2 禁止的样式写法

```tsx
// ❌ 禁止：行内 style 属性（动态样式除外）
<div style={{ marginTop: '16px' }}>...</div>

// ❌ 禁止：独立的 CSS 文件
<div className="invoice-card">...</div>

// ✅ 正确：使用 Tailwind 类
<div className="mt-4">...</div>
```

- **例外**：CSS 变量（`--color-primary`）用于动态主题时允许使用 `style` 属性

## 7. Git 工作流

### 7.1 分支策略

> 独立开发者场景，采用简化 Trunk-Based 策略。

```
main ─────●────●────●────●───
           \  /    \  /
feature    ●─●      ●─●
```

| 规则 | 说明 |
|------|------|
| **主分支** | `main`，始终可部署 |
| **功能分支** | 从 `main` 创建，完成后合并回 `main` |
| **分支生命周期** | 分支存活不超过 3 天，避免长周期分支 |
| **合并方式** | Squash Merge（保持 main 历史干净） |

### 7.2 分支命名规范

```
<type>/<short-description>
```

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | `feat/invoice-upload` |
| `fix/` | Bug 修复 | `fix/ocr-timeout` |
| `refactor/` | 重构 | `refactor/database-layer` |
| `docs/` | 文档更新 | `docs/api-spec` |
| `chore/` | 构建/配置 | `chore/setup-eslint` |

### 7.3 提交信息规范 (Conventional Commits)

所有 Git 提交信息**必须**遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<type>(<scope>): <subject>

[body]

[footer]
```

#### 类型 (type)

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变功能） |
| `style` | 代码格式调整 |
| `docs` | 文档变更 |
| `test` | 测试相关 |
| `chore` | 构建/工具/依赖 |
| `perf` | 性能优化 |

#### 作用域 (scope)

| scope | 对应范围 |
|-------|---------|
| `ocr` | OCR 识别相关 |
| `invoice` | 发票 CRUD |
| `batch` | 报销批次 |
| `excel` | 台账导出 |
| `pdf` | PDF 导出 |
| `auth` | 用户认证 |
| `reimbursement` | 费用报销单 |
| `ui` | 前端 UI 组件 |
| `api` | API 路由定义 |
| `db` | 数据库模型/迁移 |
| `config` | 项目配置 |

#### 提交信息规则

- `subject`：英文小写，**必须**以动词原形开头，不超过 50 个字符，末尾不加句号
- `body`：解释**为什么**做这个改动，不是**做了什么**，每行不超过 72 个字符
- `footer`：如果有关联 issue，使用 `Closes #123` 格式

#### 示例

```
feat(ocr): add invoice field extraction with PaddleOCR

PaddleOCR extracts invoice number, amount, date and vendor from
uploaded images. Results are returned as structured OcrResult
for user verification before persisting to database.

Closes #12
```

```
fix(batch): correct total amount calculation for substitute tickets

Substitute ticket amounts were being double-counted in batch
totals because the join query didn't filter on is_substitute flag.
Added WHERE clause to exclude substitute entries from sum.

Closes #18
```

## 8. 代码注释规范

### 8.1 何时写注释

| 需要注释 | 不需要注释 |
|----------|-----------|
| 复杂的业务逻辑（如替票多对一合并规则） | 简单的 getter/setter |
| 为什么选择某个算法/方案 | 自解释的代码（如 `if (age >= 18)`） |
| 边界条件和特殊处理 | 显而易见的类型声明 |
| API 的行为说明 | 翻译代码在做什么 |

### 8.2 注释风格

```python
# ✅ 解释 "为什么"
# PaddleOCR 对旋转图片的识别率下降 40%，所以在上传时强制正置
image = rotate_to_upright(image)

# ❌ 描述 "做了什么"（代码本身已经说清楚了）
# 旋转图片到正向
image = rotate_to_upright(image)
```

```tsx
// ✅ 解释 "为什么"
// 使用尾递归而非循环，因为发票列表可能达到千级规模
function processInvoices(list: Invoice[]): Result[] { ... }
```

```python
# TODO(username): 2026-06-01 发票查重逻辑（按发票号去重）
# FIXME: 当发票金额超过 99999.99 时，台账 Excel 单元格格式异常
```

## 9. 错误处理规范

### 9.1 服务端错误处理

统一错误返回格式：

```python
# server/app/api/ 中的统一响应
from pydantic import BaseModel

class ErrorResponse(BaseModel):
    code: str        # 机器可读的错误码，如 "INVOICE_NOT_FOUND"
    message: str     # 人类可读的错误信息，如 "发票不存在"
    detail: str | None = None  # 可选详细信息（不暴露堆栈）
```

```python
# ✅ 正确 — 使用 HTTPException
from fastapi import HTTPException

if not invoice:
    raise HTTPException(
        status_code=404,
        detail={"code": "INVOICE_NOT_FOUND", "message": "发票不存在"},
    )
```

### 9.2 客户端错误处理

- 所有 API 调用必须包裹 `try/catch`，在 UI 上显示友好的错误提示
- 使用 Zustand store 的 `error` 字段集中管理错误状态
- 网络错误显示"网络连接失败，请稍后重试"

```tsx
// ✅ 正确
try {
  await createInvoice(file);
} catch (error) {
  if (axios.isAxiosError(error)) {
    setError(error.response?.data?.message || '请求失败');
  } else {
    setError('发生未知错误');
  }
}
```

### 9.3 安全原则

- 永远不在 UI 中暴露原始错误堆栈或数据库错误信息
- 表单校验双重执行：React Hook Form 客户端校验 + FastAPI Pydantic 服务端校验
- 文件上传大小限制在前端（axios 拦截器）和后端（FastAPI `UploadFile` 限制）双重校验

## 11. 依赖管理

### 11.1 安装规范

```bash
# ✅ 正确 — 前端
npm install <package>

# ✅ 正确 — 后端
pip install <package> && pip freeze > server/requirements.txt

# ❌ 禁止 — 使用 npm install --save-exact（除非特定版本需要）
npm install <package>@<specific-version>

# ❌ 禁止 — 直接修改 requirements.txt，必须通过 pip install 生成
```

### 11.2 依赖审查

- 引入新依赖前评估：是否有活跃维护？NPM 周下载量 > 1000？是否与项目许可证兼容？
- 优先选择 TypeScript 原生类型支持良好的库（内置 `@types/*` 或自包含类型）
- 两条代码能解决的问题不引入第三方依赖

### 11.3 版本锁定

- `package-lock.json` 和 `requirements.txt` 必须提交到 Git 仓库
- 锁定文件内从无需手动修改，由包管理器自动管理
- 升级依赖使用 `npm update` / `pip install --upgrade`，然后重新生成锁定文件

## 12. 环境变量规范

### 12.1 命名规则

| 前缀 | 暴露范围 | 示例 |
|------|---------|------|
| `VITE_` | 客户端可访问 | `VITE_API_BASE_URL=http://localhost:0220` |
| 无前缀（后端） | 仅服务端 | `JWT_SECRET=your-secret-key` |
| 无前缀（后端） | 仅服务端 | `DATABASE_URL=sqlite:///./data/easybx.db` |

### 12.2 文件管理

| 文件 | 用途 | Git 追踪 |
|------|------|----------|
| `.env.example` | 环境变量模板，含所有变量名和空值占位 | ✅ |
| `.env` | 实际环境变量配置 | ❌ |

### 12.3 安全红线

- **绝对禁止**将 `JWT_SECRET`、`DATABASE_URL`、API Key 等密钥硬编码在代码中
- **绝对禁止**将 `.env` 文件提交到 Git
- 新增环境变量时**必须同步更新** `.env.example`
- 客户端环境变量必须使用 `VITE_` 前缀，无前缀的变量不会暴露给前端

## 13. AI 协作协议

### 13.1 AI 写代码前必读

| 文档 | 目的 |
|------|------|
| `specs/产品概述.md` | 理解业务需求、核心板块和验收标准 |
| `specs/技术栈.md` | 确认使用的框架、库和版本 |
| `specs/项目结构.md` | 理解目录结构和文件放置规则 |
| `specs/开发规范.md` | 遵循代码风格、命名约定和提交规范 |
| 相关模块的现有代码 | 理解当前实现，保持一致性 |

### 13.2 AI 写代码的规则

| 规则 | 说明 |
|------|------|
| **读后写** | 修改文件前必须先读取该文件的完整内容 |
| **最小改动** | 只修改任务要求的部分，不做"顺便优化"或重构 |
| **不猜版本** | 安装依赖时使用包管理器获取最新版，不手写版本号 |
| **不引入新依赖** | 除非任务明确要求 |
| **不写无意义注释** | 不写复述代码的注释（如 `// 获取用户列表` `getUsers()`） |
| **不创建多余文件** | 不主动创建 README、文档或配置文件 |
| **遵循现有模式** | 相同类型的代码必须遵循已有文件的写法和风格 |
| **提交信息规范** | 严格按照 Conventional Commits 格式写提交信息 |
| **前后端一致** | 前端 API 调用路径必须和后端路由路径一致 |

### 13.3 AI 写完代码后必做

| 步骤 | 操作 |
|------|------|
| **1. 类型检查** | 前端：`npm run typecheck` / 后端：`mypy .` |
| **2. Lint 检查** | 前端：`npm run lint` / 后端：`ruff check .` |
| **3. 依赖方向** | 确认没有违反 `specs/项目结构.md` 中的模块依赖方向 |
| **4. 环境变量** | 如有新增，同步更新 `.env.example` |
| **5. API 一致性** | 前端调用的 API 路径是否与后端定义一致 |

### 13.4 错题本机制

当 AI 生成的代码出现问题时，在 `docs/错题本.md` 中记录：

| 记录项 | 说明 |
|--------|------|
| **错误类型** | 类型错误 / 逻辑错误 / 规范违规 / 依赖错误 |
| **错误描述** | 具体出现了什么问题 |
| **根因分析** | 为什么会犯这个错 |
| **正确做法** | 下次应该怎么做 |

## 15. 安全编码规范

| 类别 | 规则 |
|------|------|
| **输入校验** | 所有用户输入必须通过 Pydantic schema（后端）和 React Hook Form 校验（前端）；文件上传校验文件类型（PDF/JPG/PNG）和大小（< 10MB） |
| **注入防护** | 禁止 SQL 拼接；所有数据库操作必须通过 SQLAlchemy ORM 方法；ORM 无法满足的场景必须使用参数化查询 |
| **XSS 防护** | React 默认转义 JSX 中的变量；禁止使用 `dangerouslySetInnerHTML` |
| **密钥管理** | 所有密钥（JWT_SECRET、API Key）放在 `.env` 文件中，代码中通过 `os.getenv()` / `import.meta.env.VITE_*` 读取 |
| **文件上传** | 文件保存使用 UUID 重命名防止路径遍历；限制文件类型为 `application/pdf`、`image/jpeg`、`image/png`；限制单文件大小不超过 10MB |
| **认证检查** | 所有非公开 API 端点必须添加 `Depends(get_current_user)` 依赖注入；JWT Token 在 axios 拦截器中自动附加到 `Authorization` 请求头 |
| **密码安全** | 密码使用 bcrypt（passlib）哈希存储，禁止明文存储；禁止在日志中输出密码 |
| **CORS 配置** | 开发环境配置 `localhost:5173`；生产环境限制为同源访问 |
