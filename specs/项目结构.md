## 3. 关键文件说明 (Key Files)

### 根目录

| 文件 | 说明 |
|------|------|
| `.env.example` | 环境变量模板：`JWT_SECRET`, `DATABASE_URL=sqlite:///../data/easybx.db` 等 |
| `.gitignore` | 忽略 `data/`、`uploads/`、`node_modules/`、`__pycache__/`、`.env` |
| `start.bat` | 一键启动脚本：先启动 uvicorn（后端），再启动 vite dev（前端开发模式） |

### 后端 (server/)

| 文件/目录 | 说明 |
|-----------|------|
| `main.py` | FastAPI 应用实例化 + 注册路由 + 挂载前端静态文件（生产模式）+ 配置 CORS |
| `app/config.py` | 从 `.env` 读取配置：`JWT_SECRET`, `DATABASE_URL`, `UPLOAD_DIR`, `OCR_ENABLED` 等 |
| `app/database.py` | SQLAlchemy `engine` + `SessionLocal` + `Base` 声明 |
| `app/models/` | ORM 模型，一个文件一个模型，清晰分离 |
| `app/schemas/` | Pydantic schema，用于请求体验证和响应序列化 |
| `app/api/` | FastAPI 路由层，只做 HTTP 相关的参数校验和响应格式，不写业务逻辑 |
| `app/services/` | **所有核心业务逻辑在此**，测试时可直接调用 service 函数 |

### 前端 (web/)

| 文件/目录 | 说明 |
|-----------|------|
| `vite.config.ts` | 配置 dev server proxy：`/api` → `http://localhost:0220` |
| `src/api/` | axios 封装，每个模块一个文件，统一管理请求地址 |
| `src/stores/` | Zustand store，管理全局状态（用户信息、发票列表、当前批次） |
| `src/pages/` | 页面组件，每个文件对应一个路由，Figma 设计稿的每个页面在此实现 |
| `src/components/` | 按业务模块分目录组织组件，`ui/` 为通用 UI 基座，与 Figma 设计系统对应 |

## 4. 模块说明 (Module Description)

### 4.1 前端页面路由映射

| 路由路径 | 页面组件 | 对应 Figma 页面 | 功能 |
|----------|----------|----------------|------|
| `/login` | LoginPage | — | 用户登录 |
| `/register` | RegisterPage | — | 用户注册 |
| `/` | DashboardPage | 概览 / 工作台 | 统计概览、最近批次、快速入口 |
| `/invoices` | InvoicesPage | 文件 | 发票列表、上传、OCR 识别、编辑 |
| `/batches` | BatchesPage | — | 报销批次列表 |
| `/batches/:id` | BatchDetailPage | — | 批次详情、发票勾选、替票、台账预览 |
| `/batches/:id/preview` | ReimbursementPreviewPage | 报销单 | 费用报销单 A4 预览、打印/导出 |

### 4.2 后端 API 端点映射

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/auth/register` | POST | 用户注册 |
| `/api/auth/login` | POST | 用户登录，返回 JWT Token |
| `/api/invoices` | GET | 获取当前用户的发票列表（支持筛选 status/日期） |
| `/api/invoices` | POST | 上传发票文件，触发 OCR 识别 |
| `/api/invoices/{id}` | GET | 获取单张发票详情 |
| `/api/invoices/{id}` | PUT | 更新发票信息（人工补全/修正） |
| `/api/invoices/{id}/archive` | POST | 单张发票归档 |
| `/api/batches` | GET | 获取批次列表 |
| `/api/batches` | POST | 创建报销批次 |
| `/api/batches/{id}` | GET | 获取批次详情（含关联发票） |
| `/api/batches/{id}` | PUT | 更新批次信息 |
| `/api/batches/{id}` | DELETE | 删除批次 |
| `/api/batches/{id}/invoices` | POST | 向批次添加发票/替票关联 |
| `/api/batches/{id}/invoices/{invoice_id}` | DELETE | 从批次移除发票 |
| `/api/batches/{id}/export-excel` | GET | 导出台账 Excel |
| `/api/exports/invoice-pdf` | POST | 合并发票导出 PDF（请求体含发票 ID 列表 + 排版设置） |
| `/api/exports/reimbursement-preview/{batch_id}` | GET | 获取费用报销单预览数据（已按事由合并） |
| `/api/exports/reimbursement-pdf/{batch_id}` | GET | 导出费用报销单 PDF |

### 4.3 后端 Service 层核心逻辑

| Service | 职责 |
|---------|------|
| `ocr_service.py` | 调用 PaddleOCR 提取发票字段 → 结构化输出（发票号/金额/日期/类别） |
| `excel_service.py` | 加载台账模版 `文件/台账模版.xlsx` → 填入数据 → 生成新 Excel |
| `pdf_service.py` | 将多张发票 PDF/图片合并 → ReportLab 布局 A4（纵向2张/横向4张）→ 输出 PDF |
| `reimbursement_service.py` | 按批次关联的发票 → 按 `expense_item`（事由）分组求和 → 生成合并数据供前端渲染 + 生成 PDF |
| `batch_service.py` | 替票关联逻辑：一对一替换 / 一对多拆分 / 多对一合并 → 在 `batch_invoices` 关联表中记录 `is_substitute` 和 `substitute_for` 字段 |

## 5. 文件放置规则 (Placement Rules)

### 5.1 后端规则

| 内容类型 | 放置位置 | 禁止 |
|----------|----------|------|
| 数据库模型 | `server/app/models/*.py` | 不要放在 api 或 services 中 |
| API 端点 | `server/app/api/*.py` | 不要在路由文件中写 SQL 或 OCR 调用 |
| 业务逻辑 | `server/app/services/*.py` | 不要直接调用 service 之外的模块 |
| Pydantic schema | `server/app/schemas/*.py` | 不要混用 ORM 模型和响应模型 |

### 5.2 前端规则

| 内容类型 | 放置位置 | 禁止 |
|----------|----------|------|
| 页面组件 | `web/src/pages/*.tsx` | 不要在页面中写复杂业务逻辑 |
| 可复用组件 | `web/src/components/[模块]/*.tsx` | 不要一次导出过多组件，一个文件一个组件 |
| API 调用 | `web/src/api/*.ts` | 不要在组件中直接写 axios 请求 |
| 状态管理 | `web/src/stores/*.ts` | 不要在其他文件中直接修改 store 外的全局状态 |

### 5.3 通用规则

- **一个文件一个职责**：不要在一个文件中定义多个不相关的模型/组件/函数
- **文件名即类名**：`user.py` → `class User`，`InvoiceCard.tsx` → `export function InvoiceCard`
- **测试文件紧邻源码**：`services/ocr_service.py` 对应的测试放在 `tests/services/test_ocr_service.py`
