## 4. API 层设计

### 4.1 `api/invoices.ts` — 完整实现

全部 9 个 API 端点的封装，响应类型使用 `types/invoice.ts` 中定义的类型：

```typescript
// api/invoices.ts — 重写

import { apiClient } from './client';
import type {
  Invoice,
  InvoiceListResponse,
  InvoiceListParams,
  UpdateInvoiceRequest,
  UploadResponse,
  DeleteResponse,
} from '@/types/invoice';

export const invoicesApi = {
  // GET /api/invoices/ — 列表（支持 state/page/page_size/date_from/date_to）
  list: async (params?: InvoiceListParams): Promise<InvoiceListResponse> => {
    const r = await apiClient.get('/invoices/', { params });
    return r.data;
  },

  // POST /api/invoices/ — 批量上传
  upload: async (files: File[]): Promise<UploadResponse> => {
    const form = new FormData();
    files.forEach((f) => form.append('files', f));
    const r = await apiClient.post('/invoices/', form, {
      headers: { 'Content-Type': 'multipart/form-data' },
    });
    return r.data;
  },

  // GET /api/invoices/{id} — 详情
  get: async (id: number): Promise<Invoice> => {
    const r = await apiClient.get(`/invoices/${id}`);
    return r.data;
  },

  // GET /api/invoices/{id}/file — 文件访问（返回 blob）
  getFileUrl: (id: number): string => {
    // 直接返回 API URL，axios 拦截器会自动添加 token
    return `/api/invoices/${id}/file`;
  },

  // PUT /api/invoices/{id} — 更新
  update: async (id: number, data: UpdateInvoiceRequest): Promise<Invoice> => {
    const r = await apiClient.put(`/invoices/${id}`, data);
    return r.data;
  },

  // POST /api/invoices/{id}/confirm — 确认入库
  confirm: async (id: number): Promise<Invoice> => {
    const r = await apiClient.post(`/invoices/${id}/confirm`);
    return r.data;
  },

  // DELETE /api/invoices/{id} — 删除
  remove: async (id: number): Promise<DeleteResponse> => {
    const r = await apiClient.delete(`/invoices/${id}`);
    return r.data;
  },

  // GET /api/invoices/trash — 回收站列表
  trashList: async (params?: { page?: number; page_size?: number }): Promise<InvoiceListResponse> => {
    const r = await apiClient.get('/invoices/trash', { params });
    return r.data;
  },

  // POST /api/invoices/{id}/restore — 恢复
  restore: async (id: number): Promise<Invoice> => {
    const r = await apiClient.post(`/invoices/${id}/restore`);
    return r.data;
  },
};
```

### 4.2 "待确认" Tab 的特殊处理

后端 `GET /api/invoices/` 不支持一次查询两个状态。有两种方案：

| 方案 | 做法 | 复杂度 |
|------|------|--------|
| A：两次请求合并 | 先请求 `state=pending`，再请求 `state=failed`，前端合并列表 | 较高 |
| B：后端支持逗号分隔（推荐） | `GET /api/invoices/?state=pending,failed` 后端解析 | 需后端改动 |
| C：前端标记 + 一次请求 | 后端不改，前端发送两次并行请求后合并 | 中等 |

**推荐方案 C**：前端在"待确认" Tab 下并行发送两个请求（`state=pending` 和 `state=failed`），按日期合并排序。理由：
- 后端不改动，零侵入
- 个人使用场景下数据量小，两次请求几乎无感
- 实现简单，`Promise.all` 即可

→ AC-003, AC-004, AC-007, AC-008, AC-012, AC-024, AC-025

**关键设计决策**：

1. **不把 `loading` 细分多个字段**。一个全局 `loading` 足够，配合 `uploading` 防止上传重复。细节加载状态由组件内部管理（如确认入库按钮的本地 loading）。
2. **`activeTab` 作为 URL query 参数同步**。页面级 `InvoicesPage` 通过 `useSearchParams` 将 `activeTab` 同步到 URL（如 `/invoices?tab=pending_failed`），store 中只维护当前值。
3. **日期筛选不触发自动请求**。用户选择日期范围后，需要点击"筛选"按钮或列表自动根据变化请求。

### 5.2 分页状态管理

### 6.2 核心组件详细设计

#### 6.2.1 InvoiceUploader（上传区）

**职责**：拖拽/点击选择文件 → 展示已选文件预览列表（含格式+大小校验标注） → 用户确认后上传

**状态**：
```typescript
interface UploaderState {
  selectedFiles: FileWithPreview[];  // 已选文件列表
  isDragOver: boolean;              // 是否正在拖拽
  uploading: boolean;               // 上传中
}

interface FileWithPreview {
  file: File;
  id: string;                       // 临时唯一标识
  isValid: boolean;                 // 格式/大小校验结果
  errorMsg: string | null;          // 校验失败原因
}
```

**校验逻辑**（纯前端，逐文件执行）：
```typescript
function validateFile(file: File): { valid: boolean; error: string | null } {
  const ext = file.name.split('.').pop()?.toLowerCase();
  if (!['jpg', 'jpeg', 'png', 'pdf'].includes(ext || '')) {
    return { valid: false, error: '不支持的文件格式' };
  }
  if (file.size > 50 * 1024 * 1024) {
    return { valid: false, error: '文件大小超过 50MB 限制' };
  }
  return { valid: true, error: null };
}
```

**交互细节**：
- 选择文件后立即校验，格式/大小不符的文件在预览列表中红色标注
- 超过 20 张时只展示前 20 张 + 提示"已超出数量限制，仅上传前 20 张"
- 点击"上传"按钮后按钮变为 loading 状态不可重复点击
- 上传完成后通过回调通知父级，切换 Tab 到"识别中"

## 7. 状态管理与数据流

### 7.1 页面级状态 vs Store 状态

| 状态 | 归属 | 同步方式 |
|------|------|---------|
| `activeTab` | URL SearchParams + Store | 初始化从 URL 读取，变化时更新 URL+Store |
| `viewMode` | Store | 本地持久化（localStorage 可选） |
| 分页信息 | Store | — |
| 日期筛选 | Store | — |
| 发票列表数据 | Store | API → Store |
| 上传流程 | InvoiceUploader 组件内部 | 组件本地 state，完成后回调 Store |
| Modal 打开状态 | 组件内部 | 关闭后刷新列表 |
| 回收站数据 | Store（trash 命名空间） | — |

### 7.2 关键数据流

**上传流程**：
```
用户拖拽文件 → Uploader 内部 selectedFiles 更新（含校验）
  → 用户点击"上传"
  → uploading=true（按钮loading）
  → store.uploadFiles(files)
    → invoicesApi.upload(files)
    → 返回 UploadResponse
  → uploading=false
  → 展示 Toast 反馈
  → store.setActiveTab('processing')
  → store.fetchInvoices()
```

**Tab/筛选/分页变化流程**：
```
用户操作（切换Tab/筛选/翻页）
  → store 状态更新
  → store.fetchInvoices()
    → 构造 InvoiceListParams（state/page/page_size/date_from/date_to）
    → invoicesApi.list(params)
    → 更新 invoices/total/totalPages
  → 组件重渲染
```

**确认入库流程**：
```
用户在 Modal 中修改字段 → 点击"确认入库"
  → 前端预校验（日期非空 + 金额>0）
  → store.confirmInvoice(id)
    → invoicesApi.confirm(id)
    → 返回已更新 Invoice
  → Toast "确认入库成功"
  → 关闭 Modal
  → store.fetchInvoices()
```

**删除流程**：
```
用户点击删除 → 确认弹窗（区分状态显示不同文案）
  → store.deleteInvoice(id)
    → invoicesApi.remove(id)
    → 返回 DeleteResponse
    → 根据 type 判断：hard → 直接移除；soft → 更新回收站角标
  → Toast 反馈
  → store.fetchInvoices()
```

## 12. 技术决策

### 决策 1："待确认" Tab 数据获取

**方案**：前端并行两次请求合并

前端在用户切换到"待确认" Tab 时，同时发送 `GET /api/invoices/?state=pending` 和 `GET /api/invoices/?state=failed`，使用 `Promise.all` 并行执行，然后合并结果按发票日期倒序排列。

**理由**：后端不改动、实现简单、个人场景数据量小。

### 决策 2：文件预览方式

**方案**：直接使用 API URL + 浏览器原生能力

图片文件 → `<img src={getFileUrl(id)} />`
PDF 文件 → `<iframe src={getFileUrl(id)} />`

`getFileUrl(id)` 返回 `/api/invoices/{id}/file` 的相对 URL，axios 拦截器会自动附加 JWT token（因为所有 `/api/` 开头的请求都会被拦截器处理）。

**不引入 `@react-pdf-viewer` 等依赖**，iframe 足够满足本阶段需求。

### 决策 3：表单校验时机

**方案**：前端只做"可确认"的预校验，后端全量校验

"确认入库"按钮的状态依赖两个表单字段：`invoice_date` 非空 + `amount > 0`。按钮的 `disabled` 状态通过 `react-hook-form` 的 `watch` 实时响应。用户修改字段时按钮状态自动变化。

实际提交时后端仍会再次校验（`PUT` 和 `POST /confirm` 都会校验），前端校验只是提升 UX，不可替代后端校验。

### 决策 4：视图模式持久化

**方案**：Store 状态 + localStorage

用户在表格/卡片之间的切换偏好持久化到 `localStorage`，刷新后保持一致。Store 初始化时从 `localStorage` 读取，切换时同步写入。

## 13. 安全与性能

**上传安全**：
- 文件格式/大小校验在前端预检，但最终由后端确定
- 上传按钮 loading 防止重复提交

**文件安全**：
- 所有文件通过 API 鉴权访问，不暴露直接文件 URL
- axios 拦截器自动附加 JWT token

**性能**：
- 表格/卡片视图只渲染当前页数据（后端分页）
- 切换 Tab/筛选时仅更新列表，不清除其他状态
- Modal 打开时再请求详情数据，不影响列表
