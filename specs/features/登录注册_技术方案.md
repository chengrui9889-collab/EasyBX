# 登录与注册 — 技术设计文档

## 1. 设计概要

**功能描述**：完善登录/注册页的交互体验与视觉设计，包括双栏布局、前端校验、Enter 提交、已登录自动跳转、注册后衔接、差异化错误提示。

**影响范围**：前端 `LoginPage`、`RegisterPage`、`App.tsx`、`api/client.ts`。无需改动后端。

**技术难点**：无。纯前端改动，不涉及后端。

**外部依赖**：依赖 Phase 1.1 已完成的后端认证 API（`POST /api/auth/login`、`POST /api/auth/register`）。

---

## 2. 架构概览

纯前端单层改动，改动范围在 4 个文件内：

1. **`api/client.ts`** — 修复 401 拦截器（认证接口放行）
2. **`App.tsx`** — 添加 `GuestRoute`（已登录拦截登录/注册页）
3. **`pages/LoginPage.tsx`** — 双栏布局 + 前端校验 + Enter + 错误区分 + 注册成功承接
4. **`pages/RegisterPage.tsx`** — 双栏布局 + 前端校验 + Enter + 错误区分 + 成功跳转传参

```
用户操作 → LoginPage / RegisterPage
              ├── 前端校验（空字段/密码长度）
              ├── authApi.login/register → API
              │     ├── 成功 → setToken / navigate('/login', { state })
              │     └── 失败 → 解析 error.response.data.detail → 映射提示文案
              └── 已登录 → GuestRoute 拦截 → navigate('/dashboard')
```

## 5. 核心逻辑

### 5.1 401 拦截器修复 → AC-008, AC-009, AC-010, AC-011

**触发条件**：axios 响应拦截器收到 401 状态码。

**当前问题**：拦截器无条件跳转 `/login`，导致登录失败时页面刷新，错误提示无法展示。

**处理流程**：
1. 检查 `error.config.url` 是否以 `/auth/login` 或 `/auth/register` 结尾
2. 若是认证接口 → 不放行跳转，直接 `return Promise.reject(error)`，让页面层处理
3. 若是其他接口（如发票列表因 token 过期） → 清除 token，`window.location.href = '/login'`

### 5.2 前端校验 → AC-006, AC-007

**触发条件**：用户点击提交按钮或按 Enter 键。

**处理流程**：
1. **登录页**：检查 `username.trim() === ''` → 设置 `errors.username = '请输入用户名'`；检查 `password === ''` → 设置 `errors.password = '请输入密码'`
2. **注册页**：同上 + 检查 `password.length < 6` → 设置 `errors.password = '密码至少需要 6 个字符'`
3. 校验失败 → `return` 不发起网络请求
4. 校验通过 → 清除 errors → 发起请求

**实现方式**：用 `useState` 维护一个 `errors` 对象 `{ username?: string; password?: string; displayName?: string }`，提交时校验。输入框的 `Input` 组件已有 `error` prop，直接传入即可。

### 5.3 错误类型映射 → AC-008, AC-009, AC-010, AC-011

**触发条件**：`catch(err)` 捕获到请求异常。

**处理流程**：
```
catch (err) {
  if (!err.response) {
    // 无响应 = 网络不通
    setError('网络连接失败，请稍后重试')
  } else if (err.response.status === 401) {
    setError('用户名或密码错误')
    setPassword('')        // 清空密码 AC-008
    passwordRef.focus()    // 焦点回到密码框 AC-008
  } else if (err.response.status === 409) {
    setError('用户名已存在')  // 不清空表单 AC-009
  } else {
    setError('服务异常，请稍后重试')
  }
}
```

### 5.4 注册成功 → 登录页衔接 → AC-004

**触发条件**：注册请求成功返回。

**处理流程**：
1. `navigate('/login', { state: { registeredUsername: username } })`
2. `LoginPage` 在 `useEffect` 中读取 `location.state.registeredUsername`
3. 若存在，设置 `username` state + 显示绿色成功提示条
4. 用 `navigate(location.pathname, { replace: true, state: {} })` 清除 state，避免刷新后重复显示

### 5.5 Enter 键提交 → AC-005

**触发条件**：用户在任一输入框中按 Enter。

**实现方式**：在 `<form>` 上保留 `onSubmit`（原生 form 行为会在子 input 按 Enter 时触发 submit 事件）。当前代码已通过 `onSubmit={handleSubmit}` 实现，本次无需额外改动。

**但**：需要确认 `Button` 是 `type="submit"`，当前代码已满足。

### 5.6 已登录拦截登录/注册页 → AC-001, AC-002

**触发条件**：已登录用户访问 `/login` 或 `/register`。

**实现方式**：在 `App.tsx` 中添加 `GuestRoute` 组件（与已有 `PrivateRoute` 相反逻辑）：

```tsx
function GuestRoute({ children }: { children: React.ReactNode }) {
  const token = useAuthStore((s) => s.token);
  if (token) return <Navigate to="/dashboard" replace />;
  return <>{children}</>;
}
```

路由包裹：
```tsx
<Route path="/login" element={<GuestRoute><LoginPage /></GuestRoute>} />
<Route path="/register" element={<GuestRoute><RegisterPage /></GuestRoute>} />
```

### 5.7 错误提示消失 → 交互规则

**触发条件**：用户开始修改任一输入框内容。

**实现方式**：在每个 `onChange` handler 中调用 `setError('')`（不清空 fields errors，因为那是提交时校验的）。

**设计语言来源**：EasyBX 现有设计语言（`bg-blue-600` 主色、`bg-gray-50` 底色、白卡片、圆角、系统字体）。

**布局结构**：

│   bg-blue-600        │   bg-white             │

## 7. 技术决策

### 决策 1：表单校验方式 — 手动 state vs React Hook Form

**背景**：技术栈中已引入 React Hook Form，但登录/注册页只有 2-3 个字段、3 条校验规则。

**选项**：
- A: React Hook Form — 项目统一技术栈，但需要改造 `Input` 组件支持 `register`，引入 `useForm` 的 boilerplate，对一个仅 3 字段的表单来说太重
- B: 手动 state + `useState<Record<string, string>>` — 轻量直白，3 行校验代码，后续大表单再用 RHF

**结论**：选 B。3 条校验规则不值得引入 RHF 的 boilerplate。后续发票信息编辑表单（多字段）再统一使用 RHF。

### 决策 2：注册成功信息传递方式 — URL query vs router state

**背景**：注册成功后需要把用户名传给登录页并显示提示。

**选项**：
- A: URL query（`/login?registered=zhangsan`）— 用 `useSearchParams` 读取，但不美观且会暴露在地址栏
- B: React Router `state`（`navigate('/login', { state })`）— 不污染 URL，state 在页面内通过 `useLocation` 读取

**结论**：选 B。不污染 URL，用户刷新登录页后提示消失也是合理的交互。

## 8. 安全与性能

**输入校验**：前端提交前校验非空和密码长度，后端也有对应的 Pydantic schema 校验。

**敏感数据处理**：密码仅在表单提交时一次性发送，不在 localStorage 或 state 中持久化。登录成功后只存 token，不存密码。
