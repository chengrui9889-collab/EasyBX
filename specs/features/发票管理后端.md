# 发票管理后端 — 功能需求说明书

## 1. 背景与价值

**为什么做**：发票是 EasyBX 的核心数据资产。用户需要将电子发票/纸质发票拍照上传到系统，由系统自动 OCR 识别出发票关键信息，待用户确认后形成结构化数据，为后续报销整理、台账生成打下基础。

**做成什么样**：用户上传发票后无需等待，系统后台异步完成 OCR 识别。识别结果出来后用户可在列表和详情页看到识别字段，手动补全缺失信息后确认入库。整个过程流畅、透明，每张发票的状态一目了然。

**关联**：`specs/开发路线图.md` Phase 1.3

---

## 2. 范围

### 本次做

- 发票文件上传（jpg/png/pdf，单文件 ≤ 50MB，单次 ≤ 20 张）
- 异步 OCR 识别（PaddleOCR），识别完成后回调更新发票记录
- 发票状态流转：processing → pending / failed → confirmed
- OCR 字段识别（通用字段 + 高铁/滴滴/飞机特色字段）
- 发票列表（按发票日期倒序、按状态筛选、分页 20/50/100/200）
- 发票详情（查看原始文件 + OCR 字段 + 手动编辑 + 报销说明）
- 确认入库（校验日期必填 + 金额 > 0）
- 发票删除：processing / pending / failed → 物理删除；confirmed → 软删除
- 回收站（30 天内可恢复，到期自动物理删除）

### 本次不做

- `archived` 归档状态（Phase 1.4 报销时处理）
- OCR 模型训练/优化（使用已有 PaddleOCR）
- 前端页面（Phase 1.4 或后续）
- 替票关联（后续 Phase）

## 4. 需求详述

### 4.1 核心流程

```
文件上传(jpg/png/pdf, 单次≤20张, 单文件≤50MB)
  │
  ├─ 先校验：格式不符 / 超50MB → 单独跳过，告知文件名+原因
  │
  ├─ 校验通过的文件 → 存服务器 (uuid_原始文件名)
  │                    发票记录 state=processing
  │                    触发异步 OCR 识别
  │
  ├─ OCR 识别成功 → state=pending
  │                 识别到的字段回填，未识别到的留空
  │
  ├─ OCR 识别完全失败 → state=failed
  │                       自动备注失败原因
  │
  └─ OCR 服务不可用/超时 → state=failed
                             自动备注原因

pending/failed 状态下：
  │
  ├─ 查看发票详情：原始图片/PDF + OCR 字段
  ├─ 手动编辑/补全字段 + 填写报销说明
  ├─ 点"确认入库"
  │   ├─ 校验：发票日期必填 + 金额 > 0 → 不通过则拒绝+提示
  │   └─ 通过 → state=confirmed
  │
  └─ 删除 → 物理删除（记录+文件）

confirmed 状态下：
  │
  ├─ 删除 → 软删除，进入回收站（30天内可恢复）
  │
  └─ 30天后 → 自动物理删除（记录+文件）
```

### 4.2 状态流转

```
         ┌──────────┐
         │processing │  OCR 识别中
         └────┬─────┘
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
┌──────┐ ┌───────┐ ┌─────────┐
│pending│ │failed │ │ (超时/) │
│ 待确认 │ │识别失败│ │服务异常) │
└──┬───┘ └───┬───┘ └────┬────┘
   │          │           │
   └──────────┼───────────┘
              │ 用户补全+确认入库
              ▼
         ┌──────────┐
         │confirmed │  已入库
         └────┬─────┘
              │ 删除
              ▼
         ┌──────────┐
         │ deleted  │  回收站 (30天)
         └────┬─────┘
              │ 30天后 / 用户手动清理
              ▼
           物理删除
```

### 4.3 文件存储规则

| 规则 | 说明 |
|------|------|
| 文件命名 | `{uuid}_{原始文件名}` |
| 存储路径 | 服务器本地目录（由配置指定） |
| 支持格式 | jpg、png、pdf |
| 单文件上限 | 50 MB |
| 单次上传上限 | 20 张 |

### 4.4 OCR 识别字段

#### 通用字段（所有发票类型共用）

| 字段 | 说明 |
|------|------|
| 发票号码 | 发票编号 |
| 开票日期 | 发票日期 |
| 金额 | 价税合计 |
| 销售方名称 | 商家/公司名称 |
| 购买方名称 | 你的公司名 |
| 发票类型 | 增值税普通/专用/出租车票/定额发票等 |

#### 特色字段（按发票类型，无则留空）

| 发票类型 | 特有字段 |
|---------|---------|
| 电子发票（增值税） | 项目名称（货物/服务名） |
| 高铁票 | 车次、出发站、到达站 |
| 滴滴打车 | 出发地、目的地 |
| 飞机行程单 | 航班号、出发地、到达地 |

### 4.5 报销说明

- 报销说明是用户手动填写的文本字段
- 在 pending / failed 状态下可编辑
- 用于备注报销事由、用途等
- 非必填

### 4.6 分页规则

| 规则 | 说明 |
|------|------|
| 默认每页 | 20 条 |
| 可选每页 | 20 / 50 / 100 / 200 |
| 排序 | 按发票日期倒序（最新在前） |
| 筛选 | 可按状态筛选（processing / pending / failed / confirmed） |

### 4.7 业务规则

| 编号 | 规则 |
|------|------|
| BR-001 | 上传文件格式限于 jpg、png、pdf，大小 ≤ 50MB |
| BR-002 | 单次最多上传 20 张，超出部分被忽略并告知用户 |
| BR-003 | 文件命名：`{uuid}_{原始文件名}` |
| BR-004 | 确认入库必要条件：发票日期必填 + 金额 > 0 |
| BR-005 | processing / pending / failed 状态删除 → 立即物理删除 |
| BR-006 | confirmed 状态删除 → 软删除，进入回收站，30 天内可恢复 |
| BR-007 | 软删除 30 天后自动物理删除（记录 + 文件） |
| BR-008 | OCR 识别以异步方式执行，上传后立即返回，不阻塞用户 |
| BR-009 | 每张发票只属于一个用户，用户只能看到自己的发票 |

## 5. 验收标准

### 5.1 正常流程 (Happy Path)

- [ ] **AC-001**: 上传单张发票文件
  - **Given** 用户已登录
  - **When** 上传一张 jpg/png/pdf 格式、大小 ≤ 50MB 的发票文件
  - **Then** 文件保存到服务器，命名格式为 `{uuid}_{原始文件名}`，数据库生成发票记录，state=processing，返回发票基本信息

- [ ] **AC-002**: 批量上传含不符合要求的文件
  - **Given** 用户已登录
  - **When** 一次上传多张发票，其中部分格式不符（如 .doc）或超过 50MB
  - **Then** 符合要求的文件正常上传并生成发票记录，不符合要求的文件被单独跳过，返回结果中明确告知哪张文件、为什么被跳过

- [ ] **AC-003**: OCR 识别成功
  - **Given** 发票 state=processing
  - **When** OCR 异步识别成功完成
  - **Then** 发票 state 更新为 pending，OCR 识别到的字段回填到对应字段（如日期、金额、销售方等），未识别到的字段保持为空

- [ ] **AC-004**: OCR 识别完全失败
  - **Given** 发票 state=processing，上传的是一张非发票图片或图片严重模糊
  - **When** OCR 无法识别出任何有效字段
  - **Then** 发票 state 更新为 failed，自动备注失败原因（如"图片质量过低，无法识别"）

- [ ] **AC-005**: 查看 pending/failed 发票详情
  - **Given** 存在 state=pending 或 failed 的发票
  - **When** 用户查看发票详情
  - **Then** 返回原始文件访问地址 + 所有字段（OCR 已识别的有值，未识别的为空）+ 报销说明

- [ ] **AC-006**: 确认入库 — 校验通过
  - **Given** 发票 state=pending 或 failed，用户已将发票日期和金额补全（日期有效、金额 > 0）
  - **When** 用户点击"确认入库"
  - **Then** 发票 state 更新为 confirmed，所有用户填写的字段（含报销说明）保存生效

- [ ] **AC-007**: 发票列表 — 默认排序与分页
  - **Given** 用户已登录，数据库中存在多张发票
  - **When** 用户查看发票列表（不传任何筛选/分页参数）
  - **Then** 返回按发票日期倒序排列的结果，默认每页 20 条

- [ ] **AC-008**: 发票列表 — 按状态筛选
  - **Given** 用户已登录，数据库中存在不同状态的发票
  - **When** 用户按状态筛选（如 `state=pending`）
  - **Then** 只返回符合该状态的发票，排序仍按发票日期倒序

- [ ] **AC-009**: 删除 processing/pending/failed 发票
  - **Given** 发票 state 为 processing / pending / failed
  - **When** 用户删除该发票
  - **Then** 数据库记录被删除，服务器上的文件被删除，发票不再出现在列表中

- [ ] **AC-010**: 删除 confirmed 发票（软删除）
  - **Given** 发票 state=confirmed
  - **When** 用户删除该发票
  - **Then** 发票标记为软删除（进入回收站），数据库记录保留，文件保留，发票从主列表中消失

- [ ] **AC-011**: 回收站中恢复发票
  - **Given** 一张 confirmed 发票被软删除，距删除时间 < 30 天
  - **When** 用户在回收站中点击恢复
  - **Then** 发票重新出现在主列表中，state 恢复为 confirmed

- [ ] **AC-012**: 更新发票字段
  - **Given** 发票 state=pending 或 failed
  - **When** 用户修改发票的任意字段（如更正 OCR 识别不准的金额、补充缺失的日期、填写报销说明）
  - **Then** 字段更新成功，发票 state 保持不变

### 5.2 边界与异常 (Edge & Error Cases)

- [ ] **AC-013**: 上传非支持格式文件
  - **Given** 用户已登录
  - **When** 上传 .doc / .txt / .exe 等非 jpg/png/pdf 格式的文件
  - **Then** 该文件被拒绝上传，返回结果中明确告知文件名和不被接受的原因"不支持的文件格式"

- [ ] **AC-014**: 上传超过 50MB 的文件
  - **Given** 用户已登录
  - **When** 上传单文件大小 > 50MB
  - **Then** 该文件被单独跳过，返回结果中明确告知文件名和原因"文件大小超过 50MB 限制"

- [ ] **AC-015**: 单次上传超过 20 张
  - **Given** 用户已登录
  - **When** 一次上传 25 张发票
  - **Then** 只处理前 20 张，返回结果中明确告知超出 20 张的部分被忽略

- [ ] **AC-016**: 确认入库 — 日期为空
  - **Given** 发票 state=pending，发票日期字段为空
  - **When** 用户点击"确认入库"
  - **Then** 拒绝操作，返回提示"请填写发票日期"

- [ ] **AC-017**: 确认入库 — 金额不合法
  - **Given** 发票 state=pending，金额字段为空、为 0 或负数
  - **When** 用户点击"确认入库"
  - **Then** 拒绝操作，返回提示"金额必须大于 0"

- [ ] **AC-018**: OCR 服务不可用或超时
  - **Given** 发票 state=processing
  - **When** OCR 服务在合理时间内无响应或返回错误
  - **Then** 发票 state 自动更新为 failed，备注栏记录失败原因（如"OCR 服务异常，请联系管理员"）

- [ ] **AC-019**: OCR 部分成功
  - **Given** 发票 state=processing，OCR 成功识别出日期和金额但发票号码模糊
  - **When** OCR 识别完成
  - **Then** 发票 state=pending，已识别的字段（日期、金额）有值，未识别的字段（发票号码）留空，用户可在详情页手动补全

- [ ] **AC-020**: 软删除 30 天后自动清理
  - **Given** 一张 confirmed 发票被软删除，且距删除时间已超过 30 天
  - **When** 系统执行定期清理任务
  - **Then** 该发票的数据库记录被物理删除，服务器文件被物理删除

### 5.3 业务规则验证 (Business Rules)

- [ ] **AC-021**: 文件命名规则 ← BR-003
  - **Given** 用户上传文件 `滴滴电子发票.jpg`
  - **When** 文件保存到服务器
  - **Then** 文件名格式为 `{uuid}_滴滴电子发票.jpg`，uuid 为唯一标识

- [ ] **AC-022**: 发票状态流转 ← BR-001, BR-004, BR-005, BR-006, BR-008
  - **Given** 任意发票
  - **When** 状态发生变化
  - **Then** 严格遵守流转规则：上传→processing，OCR 成功→pending，OCR 失败→failed，确认入库→confirmed，删除 confirmed→软删除

- [ ] **AC-023**: 分页参数 ← BR-002
  - **Given** 用户查看发票列表
  - **When** 传入 `page_size` 参数
  - **Then** 支持 20 / 50 / 100 / 200 四种分页大小，不合法参数回退到默认 20

- [ ] **AC-024**: 数据隔离 ← BR-009
  - **Given** 用户 A 和用户 B 分别上传了发票
  - **When** 用户 A 查看发票列表
  - **Then** 只能看到用户 A 自己的发票，看不到用户 B 的发票
