# 费用报销单预览 — 技术设计文档

## 1. 设计概要

**功能描述**：用户从批次详情页一键跳转到 A4 纸风格报销单预览页，系统读取批次台账数据，按发票类别合并、金额求和，生成大写金额，前端渲染为传统纸质报销单的格位格式。

**影响范围**：后端 `exports` 模块（API + Schema + Service）、前端 `ReimbursementPreviewPage` + `BatchDetailPage` + 工具函数

**技术难点**：金额拆分到传统格位的精确渲染（百十万千百十元角分 7 位）、固定 6 行/页的续页逻辑、中文大写金额的完整转换

**外部依赖**：依赖已完成的 `Phase 1.5 批次管理`（`ReimbursementBatch` + `BatchInvoice` 数据模型）

---

## 2. 架构概览

本功能属于**中等复杂度**：跨 2 个后端模块（exports API + 新建 reimbursement service），1 个前端页面 + 1 个入口改动。


**数据流**：
1. 前端携带 `batch_id` 请求 API
2. API 校验用户权限（`get_current_user`）后调用 service
3. Service 查询批次 + 台账行 + 关联发票，按类别名合并求和
4. Service 将合计金额转为中文大写
5. 返回 JSON 给前端
6. 前端纯渲染：A4 布局 + 格位拆分 + 续页分页

## 3. 数据库设计

**本次无需新增表，无需修改现有表。** 所有数据来自已有的三张表：

| 表 | 用到的字段 |
|----|-----------|
| `reimbursement_batches` | `department`, `report_date`, `reporter` |
| `batch_invoices` | `invoice_id`, `expense_item`, `source_type` |
| `invoices` | `category`, `amount` |

合并 key 的取值逻辑（后端 Service 层实现，不落库）：
- `source_type = "invoice"` → 取关联 `Invoice.category` 作为分组键
- `source_type = "manual"` → 取 `BatchInvoice.expense_item` 作为分组键
- 相同分组键名称的行合并为一条，金额累加求和

## 5. 核心逻辑

### 5.1 按类别合并分组 → AC-003, AC-016, AC-017

**触发条件**：`GET /api/exports/reimbursement-preview/{batch_id}` 被调用

**处理流程**：
1. 查询批次基本信息（department, report_date, reporter）
2. 查询批次下所有台账行（BatchInvoice），排除替票行（`is_substitute == True and source_type == "invoice"`）
3. 遍历每行确定分组键：发票行取 `Invoice.category`，手动行取 `BatchInvoice.expense_item`
4. 取金额：发票行取 `Invoice.amount`，手动行取 `BatchInvoice.row_amount`
5. 按分组键分组 → `dict[str, float]`
6. 求和后构造 `items` 列表，按金额降序排列
7. 计算合计并转为中文大写

**伪代码**：
```python
def get_reimbursement_preview(db, user_id, batch_id):
    batch = query(ReimbursementBatch).filter(id=batch_id, user_id=user_id).first()
    if not batch:
        raise 404
    
    batch_invoices = query(BatchInvoice).filter(
        batch_id=batch_id,
        # 排除替票行
        or_(is_substitute==False, substitute_for.isnot(None))
    ).all()
    
    if not batch_invoices:
        raise 400("批次中没有发票数据")
    
    groups = {}  # key → total_amount
    for bi in batch_invoices:
        key = None
        amount = 0.0
        if bi.source_type == "invoice" and bi.invoice_id:
            invoice = query(Invoice).filter(id=bi.invoice_id).first()
            key = invoice.category if invoice else "其他"
            amount = invoice.amount if invoice else 0.0
        elif bi.source_type == "manual":
            key = bi.expense_item or "其他"
            amount = bi.row_amount or 0.0
        
        if key:
            groups[key] = groups.get(key, 0.0) + amount
    
    total = sum(groups.values())
    return ReimbursementPreviewResponse(
        department=batch.department,
        report_date=batch.report_date,
        reporter=batch.reporter,
        items=[{"expense_item": k, "amount": round(v, 2)} for k, v in sorted(groups.items())],
        total_amount=round(total, 2),
        total_amount_cn=amount_to_chinese(total),
    )
```

### 5.2 中文大写金额转换 → AC-007

**实现位置**：`services/reimbursement_service.py` 中的 `amount_to_chinese()` 函数

**处理流程**：
1. 输入 `float` 金额（如 `468.00`）
2. 分离整数部分和小数部分
3. 整数部分逐位映射（0-9 → 零壹贰叁肆伍陆柒捌玖）+ 单位（元拾佰仟万）
4. 角分部分：角位填对应大写数字 + "角"，分位填对应大写数字 + "分"
5. 整数为 0 且角分为 0 → "零元整"
6. 整数部分非零且角分均为 0 → "{大写整数}元整"
7. 整数部分非零且有角分 → "{大写整数}元{大写角分}"

**示例**：
| 输入 | 输出 |
|------|------|
| 468.00 | 肆佰陆拾捌元整 |
| 0.00 | 零元整 |
| 100.50 | 壹佰元伍角整 |
| 0.30 | 零元叁角整 |

### 5.3 金额格位拆分 → AC-004, AC-006, AC-012, AC-013

**实现位置**：前端 `lib/utils.ts` 中的 `splitAmountToDigits(amount: number): (string|null)[]` 函数

**处理流程**：
1. 输入金额（如 `168.00`）
2. 格式化为 `"000016800"`（百万/十万/万/千/百/十/元 + 角分共 9 位）
3. 每位映射到数组中：
   - `[0]` 百万位, `[1]` 十万位, `[2]` 万位, `[3]` 千位, `[4]` 百位, `[5]` 十位, `[6]` 元位
   - `[7]` 角位, `[8]` 分位 — 仅用于渲染，不在报销单格位中展示（传统表单中角分位于元位右侧且不单独标注列头）
4. 前导零返回 `null`（前端渲染为空格），个位及之后返回字符串

### 5.4 固定 6 行续页 → AC-010, AC-011, AC-015

**实现位置**：前端 `ReimbursementPreviewPage.tsx`

**处理流程**：
1. 从 API 获取 `items: [{expense_item, amount}]` 列表
2. 按每页 6 行切片：`pages = chunk(items, 6)`
3. 每页计算本页合计（独立求和）
4. 渲染 N 张 A4 纸，每张包含：表头 + 6 行数据区 + 合计 + 大写 + 签名区
5. 数据不足 6 行时，剩余行保持空白格子

## 7. 技术决策

### 7.1 合并逻辑放后端还是前端

**背景**：合并（group by + sum）可以用后端 Python 实现，也可以返回原始台账行让前端用 JavaScript 实现。

**选项**：
- A: 后端合并 — 网络传输数据量小，逻辑集中在后端，前后端职责清晰
- B: 前端合并 — API 复用现有 `get_batch_detail`，不需要新接口

**结论**：选 A。本功能已有独立的预览页和路由，需要专用的 API。且中文大写转换更适合在 Python 侧实现（Python 的 decimal + 字符串处理更可靠），将合并逻辑一起放在后端保持数据处理的原子性。

### 7.2 中文大写在哪个层实现

**背景**：金额转中文大写是纯数据转换，前后端都能做。

**选项**：
- A: 后端实现 — 一次转换，前端直接用；便于测试
- B: 前端实现 — 减少后端代码，但每个客户端都要执行一次

**结论**：选 A。后端一次性转换，前端的 `amountToChinese()` 仅用于每页独立合计的大写（续页时每页合计不同，由前端就地计算）。两端各有一个版本，但后端的是"权威"版本（整批合计），前端的是辅助版本（单页合计）。

### 7.3 续页时每页合计如何计算

**背景**：AC-015 要求每一页都有独立的合计。这涉及纯前端逻辑。

**选项**：
- A: 前端按 `items.slice(page * 6, (page + 1) * 6)` 就地求和
- B: 后端按页切分后返回二维数组

**结论**：选 A。数据来自同一个 API 响应，前端分页纯属 UI 布局，不需要后端参与。每页合计 = `items.slice(...).reduce(sum, 0)`，每页大写调用前端的 `amountToChinese()`。

## 8. 安全与性能

**输入校验**：`batch_id` 为路径参数，FastAPI 自动校验类型（int）。Service 层校验批次归属（`user_id` 匹配），防止越权访问。

**性能考量**：
- 单批次发票数量通常 ≤ 50 张，类别数 ≤ 10，查询和分组在 O(n) 内完成
- API 响应大小约 1-2 KB，无性能瓶颈
- 前端续页渲染：最多 2-3 页 A4 纸（18 个类别以内），DOM 节点数可控
