# 发票归档系统 — 功能需求说明书

## 1. 背景与价值

**为什么做**：用户完成报销批次后，已入库的发票仍会一直显示在"已入库"列表中，随着报销次数增多，列表越来越臃肿，难以区分哪些已报销、哪些待报销。归档功能让用户可以将已完成报销的发票从活跃视图中"退休"，保持发票列表清爽。

**做成什么样**：用户创建一个报销批次 → 勾选发票 → 台账编辑 → 导出台账 → 点"完成批次"锁定 → 点"归档这批发票" → 批次内所有发票从"已入库"移至"已归档"视图。用户可在"已归档"视图查看归档发票，也支持单张恢复或整批撤销归档。

**关联**：
- `specs/开发路线图.md` Phase 3.1
- `specs/features/报销批次管理.md`（上游依赖）

## 2. 范围

### 本次做

- 批次"完成"操作（draft → completed，锁定编辑）
- 批次"归档"操作（completed → archived，批量归档批次内所有发票）
- 归档确认弹窗（提示发票数量，用户确认后执行）
- 发票管理页新增"已归档" Tab
- 归档发票单张恢复（在"已归档" Tab 中恢复某张发票到 confirmed）
- 批次"撤销归档"（在已归档批次详情页整批撤销归档）
- 批次列表增加"已归档"筛选
- 已归档批次只读详情页

### 本次不做

| 功能 | 理由 |
|------|------|
| 自动归档（定时任务） | 只有手动触发 |
| 归档记录的物理删除/清理 | 归档即永久保留 |
| 归档通知 | 无需通知 |
| 归档统计（概览页） | 未来扩展 |
| 归档发票的高级搜索/筛选 | 与现有 Tab 一致即可 |

## 4. 需求详述

### 4.1 核心流程

```
批次创建(draft)
  → 勾选发票、台账编辑、导出台账
  → 用户点"完成批次"
      → 批次状态 = completed
      → 锁定：不可增减发票、不可修改台账
      → 仍可：导出台账、导出 PDF
  → 用户点"归档这批发票"
      → 弹出确认弹窗："归档后，该批次的 N 张发票将从已入库移至归档区"
      → 用户确认
      → 批次状态 = archived
      → 批次内所有发票状态 = archived
      → 发票从"已入库"消失，出现在"已归档" Tab
```

### 4.2 批次完整生命周期

```
draft(草稿) ──"完成批次"──→ completed(已完成) ──"归档"──→ archived(已归档)
                                    ↑                        │
                                    │──"撤销归档"─────────────┘
                                    │
                              "单张恢复"→ 发票回到 confirmed
```

| 状态 | 可编辑台账 | 可增减发票 | 可导出 | 可归档 | 可撤销归档 | 可单张恢复 |
|------|-----------|-----------|--------|--------|-----------|-----------|
| draft | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| completed | ❌ | ❌ | ✅ | ✅ | ❌ | ❌ |
| archived | ❌ | ❌ | ✅ | ❌ | ✅ | ✅（在已归档视图） |

### 4.3 归档确认弹窗

- **触发**：批次详情页中，批次状态为 completed 时，显示"归档这批发票"按钮
- **内容**：
  ```
  确定归档该批次吗？
  归档后，该批次的 {N} 张发票将从已入库移至归档区，
  您仍可在"已归档"视图中查看和管理它们。
  [取消]  [确定归档]
  ```
- **按钮规则**：批次内发票数为 0 时，"归档这批发票"按钮置灰，提示"该批次无发票，无需归档"

### 4.4 已归档 Tab（发票管理页）

- **位置**：在现有 Tab 栏中新增，顺序为"全部 / 识别中 / 待确认 / 已入库 / **已归档**"
- **数据范围**：只展示 state=archived 的发票
- **展示规则**：与"已入库" Tab 一致的卡片/表格展示，排序按发票日期倒序
- **操作差异**：
  - 查看详情 Modal：只读模式，不可编辑任何字段
  - 恢复按钮：点击后恢复该发票为 confirmed 状态
  - 不可删除
- **空状态**："暂无归档发票，完成报销批次后归档即可查看"

### 4.5 已归档批次详情页（只读）

- **入口**：批次列表点击已归档批次卡片
- **页面模式**：所有表单字段置灰只读
- **可查看内容**：批次基本信息（部门、期间、报账人等）、关联发票列表、台账表格数据
- **可操作**：导出台账 Excel
- **特殊操作**："撤销归档"按钮
  - 弹出确认弹窗："撤销归档后，该批次 {N} 张发票将恢复为已入库状态，批次恢复为已完成。确定撤销吗？"
  - 点击确认后：批次状态回退到 completed，所有 archived 发票恢复为 confirmed

### 4.6 批次列表筛选

- **筛选项**：在现有批次列表筛选条件中增加"已归档"选项
- **展示**：已归档批次卡片显示"已归档"标签
- **数据隔离**：用户只能看到自己的批次，与现有规则一致

### 4.7 业务规则

| 编号 | 规则 |
|------|------|
| BR-001 | 只有 completed 状态的批次才能执行归档操作 |
| BR-002 | 归档操作一次性将批次内所有发票标记为 archived |
| BR-003 | 批次内发票数为 0 时，不允许归档 |
| BR-004 | 归档后发票不再出现在发票选择器（添加发票弹窗）的可选列表中 |
| BR-005 | 单张恢复的发票状态变为 confirmed，不影响所属批次的状态 |
| BR-006 | 撤销归档时，只恢复批次中仍为 archived 状态的发票（已被单张恢复的不受影响） |
| BR-007 | 已归档批次详情页为只读，不可编辑任何内容 |
| BR-008 | 归档后的批次仍可导出台账 Excel |

## 5. 验收标准

### 5.1 正常流程 (Happy Path)

- [ ] **AC-001: 完成批次 — 锁定编辑**
  - **Given** 批次状态为 draft，已关联 5 张发票，台账已编辑
  - **When** 用户点击"完成批次"
  - **Then** 批次状态变为 completed，不得再添加/移除发票，不得修改台账表格，但导出台账和导出 PDF 仍然可用

- [ ] **AC-002: 归档批次 — 确认弹窗**
  - **Given** 批次状态为 completed，关联了 5 张发票
  - **When** 用户点击"归档这批发票"
  - **Then** 弹出确认弹窗"归档后，该批次的 5 张发票将从已入库移至归档区。确定归档吗？"，用户确认后执行归档

- [ ] **AC-003: 归档批次 — 执行成功**
  - **Given** 用户确认了归档弹窗
  - **When** 归档操作执行完成
  - **Then** 批次状态变为 archived，该批次内所有发票状态变为 archived，页面提示"归档成功"

- [ ] **AC-004: 已归档视图 — 查看归档发票**
  - **Given** 发票管理页面上存在已归档发票
  - **When** 用户点击"已归档" Tab
  - **Then** 列表只展示 state=archived 的发票，展示内容与"已入库" Tab 一致（发票类型图标、金额、日期、销售方、状态标签），排序按发票日期倒序

- [ ] **AC-005: 已归档批次 — 只读详情页**
  - **Given** 批次状态为 archived
  - **When** 用户点击批次卡片进入详情页
  - **Then** 页面为只读模式，所有字段置灰不可编辑，可查看批次信息、关联发票列表、台账数据，可导出台账 Excel

- [ ] **AC-006: 单张恢复归档发票**
  - **Given** "已归档" Tab 中有一张 archived 状态的发票
  - **When** 用户点击该发票的"恢复"按钮并确认
  - **Then** 发票状态从 archived 恢复为 confirmed，从"已归档" Tab 消失，重新出现在"已入库" Tab

- [ ] **AC-007: 整批撤销归档**
  - **Given** 批次状态为 archived，关联了 5 张发票
  - **When** 用户在批次详情页点击"撤销归档"并确认
  - **Then** 批次状态恢复为 completed，5 张发票状态全部从 archived 恢复为 confirmed，页面提示"撤销归档成功"

- [ ] **AC-008: 批次列表 — 已归档筛选**
  - **Given** 批次列表中存在已归档批次
  - **When** 用户选择筛选条件"已归档"
  - **Then** 列表只展示 status=archived 的批次，每个批次卡片展示"已归档"标签

### 5.2 边界与异常 (Edge & Error Cases)

- [ ] **AC-009: 归档空批次 — 不允许操作**
  - **Given** 批次状态为 completed，但未关联任何发票
  - **When** 用户点击"归档这批发票"
  - **Then** 按钮置灰不可点击，或点击后提示"该批次无发票，无需归档"

- [ ] **AC-010: 撤销归档时部分发票已被单张恢复**
  - **Given** 批次 X 归档了 5 张发票，其中 1 张已被用户单张恢复为 confirmed
  - **When** 用户对批次 X 执行"撤销归档"
  - **Then** 剩余 4 张 archived 状态的发票恢复为 confirmed，批次状态恢复为 completed，已恢复的那张发票不受影响

- [ ] **AC-011: 已归档 Tab 空状态**
  - **Given** 从未归档过任何发票
  - **When** 用户切换到"已归档" Tab
  - **Then** 展示空状态："暂无归档发票，完成报销批次后归档即可查看"

- [ ] **AC-012: 已归档发票详情 — 只读模式**
  - **Given** "已归档" Tab 中有一张发票
  - **When** 用户点击查看详情
  - **Then** Modal 弹窗打开，左侧可预览原文件，右侧表单全部字段置灰只读，无保存/确认入库/删除按钮，仅有"关闭"和"恢复"按钮

- [ ] **AC-013: 已归档批次详情 — 撤销归档确认弹窗**
  - **Given** 批次状态为 archived，关联 5 张发票
  - **When** 用户点击"撤销归档"
  - **Then** 弹出确认弹窗"撤销归档后，该批次 5 张发票将恢复为已入库状态，批次恢复为已完成。确定撤销吗？"，确认后执行

### 5.3 业务规则验证 (Business Rules)

- [ ] **AC-014: 批次状态流转规则 ← BR-001**
  - **Given** 任意批次
  - **When** 状态发生变化
  - **Then** 严格遵守流转规则：draft → completed → archived，不允许 draft 直接到 archived

- [ ] **AC-015: 数据隔离 ← BR-002, BR-008**
  - **Given** 用户 A 和用户 B 各自有归档发票和批次
  - **When** 用户 A 查看"已归档" Tab 和批次列表
  - **Then** 只能看到用户 A 自己的归档发票和批次

- [ ] **AC-016: 归档发票不出现在选择器中 ← BR-004**
  - **Given** 发票 A 已被归档
  - **When** 用户打开任意批次的发票选择器
  - **Then** 发票 A 不在可选列表中（只有 confirmed 的发票才可选）

- [ ] **AC-017: 归档后"已入库" Tab 数量更新**
  - **Given** "已入库" Tab 角标显示 10 张发票
  - **When** 归档一个包含 3 张发票的批次后
  - **Then** "已入库" Tab 角标变为 7，"已归档" Tab 角标显示 3（如支持角标）

| `GET /api/batches/{id}` | 已归档批次只读详情 | AC-005 |
