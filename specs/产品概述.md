# 产品概述

## 1. 项目愿景 (Vision)
*用一句话概括：我们要为谁，提供什么核心价值？*

> 为需要频繁报销的企业员工，提供从发票上传 OCR 识别 → 台账自动生成 → PDF 发票整理 → 费用报销单预览的一站式智能报销管理工具，告别手动整理发票和手写报销单的低效流程。

## 2. 核心价值 (Value Proposition)
*   **用户价值**：将传统线下报销中"拍照→打印→贴票→手写报销单→计算金额"的繁琐流程，缩减为"上传发票→OCR 自动识别→勾选生成台账→一键预览报销单"，单次报销耗时从小时级降低到分钟级。
*   **商业/技术价值**：采用 PaddleOCR 离线识别方案，零 API 调用费用；个人级轻量部署（SQLite + FastAPI），无需运维复杂的基础设施；支持替票业务（一对一、一对多、多对一），覆盖真实报销场景的痛点。

## 3. 核心板块 (Core Modules)
*将系统拆解为关键的顶层模块。*

### 3.1 发票管理模块
*   **描述**：负责发票的全生命周期管理 —— 上传、识别、校验、补全、归档。是所有报销流程的数据入口。
*   **关键特性**：
    *   支持电子发票 PDF 和纸质发票拍照图片（JPG/PNG）上传
    *   PaddleOCR 自动识别发票关键字段：发票号码、金额、日期、事由/类别、抬头等
    *   识别结果支持人工校验和手动修正补全
    *   已报销发票自动归档，按报销批次分类存储，从活跃视图移除

### 3.2 报销管理模块
*   **描述**：负责报销批次的创建、发票勾选关联、替票处理，以及按照台账模版格式生成 Excel 台账。
*   **关键特性**：
    *   创建报销批次：配置报销部门、报销期间、报账人、收款人、银行卡号等完整信息
    *   从已识别发票列表中勾选发票加入批次
    *   替票管理：先上传替票发票 → 关联到费用行 → 台账备注自动标注替票信息；支持一对一替换、一对多拆分、多对一合并三种模式
    *   按台账模版自动生成 Excel 台账（含日期、事由、数量、单价、金额、垫款金额、备注、合计行、审核人/报账人签名区）

### 3.3 导出与预览模块
*   **描述**：负责报销成果物的可视化预览与导出，包括发票合并 PDF 和费用报销单预览。
*   **关键特性**：
    *   多张发票合并导出为可打印 PDF，支持 A4 纵向（一页 2 张）和 A4 横向（一页 4 张）两种排版
    *   费用报销单预览：按事由合并展示（相同事由合并为一行，金额求和），A4 纸风格页面呈现
    *   预览页面支持直接打印和导出 PDF
    *   UI 依据 Figma 设计稿实现（左侧边栏导航 + 顶部工具栏 + A4 纸预览区域）

## 4. 目标用户 & 核心场景 (User & Scenes)
### 4.1 目标用户
*   **企业员工（报账人）**：需要频繁因公出差或采购后进行报销的个人用户。核心诉求是将零散的发票快速整理为规范的报销材料，减少手动填单的出错和时间消耗。
*   **财务审核人（轻度使用）**：可查看生成的台账和报销单，核对发票与费用的一致性。本系统不做审批流，审核为线下行为。

### 4.2 核心场景 (Key Use Cases)
*   **场景 1 — 发票入库**：员工出差归来，手上有高铁票（拍照图片）、打车电子发票（PDF）和餐饮票，逐一上传到系统，OCR 自动识别金额/日期/类别，员工快速核对后确认入库。
*   **场景 2 — 报销批次整理**：员工创建 12 月报销批次，填写报销部门和期间，从发票库中勾选本次需报销的发票，系统自动按台账模版生成 Excel 台账。
*   **场景 3 — 替票处理**：某笔"奖金"费用无发票可开，员工从发票库中选择一张金额匹配的交通票作为替票，关联到费用行，台账备注自动标注"替票"信息。
*   **场景 4 — 打印报销单**：员工查看费用报销单预览（已按事由合并），确认无误后直接打印或导出 PDF，连同发票合并 PDF 一起提交给财务。

## 5. 关键业务流程 (Core Workflows)
*描述系统最核心的业务链路，这决定了技术架构的复杂度。*

**主流程 — 发票入库链路：**

```
用户上传文件(PDF/JPG/PNG) → 后端文件接收与格式校验
  → PaddleOCR 发票识别 → 提取字段(发票号/金额/日期/事由等)
  → 返回识别结果给前端 → 用户人工校验/补全
  → 确认后入库(存入数据库，标记"未报销"状态)
```

**主流程 — 报销台账生成链路：**

```
用户创建报销批次(部门/期间/报账人等) → 从发票库勾选发票
  → (可选)替票关联：选替票→绑定费用行→生成替票备注
  → 系统按台账模版排版 → 生成 Excel 台账文件
  → 按事由合并费用 → 生成费用报销单预览页面
  → 用户确认后，导出 PDF / 打印
  → 发票状态标记为"已归档" → 归入当前批次分类存储
```

## 6. 验收标准 (Acceptance Criteria)
*项目第一阶段结束的具体标志是什么？（用于防止需求蔓延）*

*   **功能完备性**：
    *   发票上传与 OCR 识别主流程跑通，核心字段识别准确率可用（支持人工修正兜底）
    *   报销批次从创建到台账 Excel 导出全链路可用
    *   替票（一对一/一对多/多对一）关联与台账备注生成正常
    *   费用报销单预览：按事由合并、A4 纸风格正确呈现
    *   多张发票合并 PDF 导出，纵向/横向排版均可用
    *   用户登录/注册功能正常
*   **技术指标**：
    *   单张发票 OCR 识别耗时 < 5 秒
    *   前端首屏加载 < 3 秒
    *   后端 API 接口无未处理的异常崩溃
*   **UI 还原**：
    *   前端页面与 Figma 设计稿关键布局一致（侧边栏导航 + A4 预览区域 + 工具栏）

## 7. 约束条件 (Constraints)
*非功能性的硬性限制。*

*   服务端口固定为 **0220**
*   个人单用户使用场景（支持多用户登录但无审批流）
*   桌面端 Web 应用，暂不做移动端适配
*   OCR 使用 PaddleOCR 离线方案，不依赖外部付费 API
*   数据库使用 SQLite（后续可迁移至 PostgreSQL）
*   API Key 等敏感配置统一放入 `.env` 文件管理

## 8. 项目边界 (Out of Scope)
*明确本阶段**不做**什么，避免需求蔓延。*

*   不实现审批工作流（审核为线下行为）
*   不做移动端适配（仅桌面端 Web）
*   替票金额与费用金额的差额不自动处理
*   不做发票查重（基于发票号去重的逻辑暂不实现）
*   不做多币种支持
*   不做与外部财务系统（如 ERP）的对接
