# 技术栈选型

## 1. 核心决策 (Key Decisions)

*简述为什么选择这套技术栈？*

| 决策 | 选择 | 理由 |
|------|------|------|
| **后端语言** | Python | OCR 识别是整个系统的核心瓶颈，PaddleOCR 是 Python 生态的库，选 Python 可以避免跨语言调用 OCR 的复杂度和性能损耗 |
| **Web 框架** | FastAPI | 异步支持优秀，原生集成 Pydantic 做数据校验，自动生成 OpenAPI 文档，开发效率高，且与 Python OCR/Excel/PDF 生态无缝衔接 |
| **前端框架** | React 18 + TypeScript | 桌面端复杂表单交互场景下，React 的组件化模式最适合 Figma 设计稿的还原；TypeScript 提供类型安全，减少发票/批次等数据模型的运行时错误 |
| **样式方案** | Tailwind CSS | 对照 Figma 设计稿实现 UI 时，Tailwind 的原子化 CSS 可以精确控制每个像素的间距/颜色/圆角，无需写自定义 CSS |
| **数据库** | SQLite | 个人使用场景，零运维成本，文件级存储方便备份，后续可无缝迁移至 PostgreSQL |

## 3. 后端 (Backend)

| 维度 | 选型 | 版本 | 说明 |
|------|------|------|------|
| **运行时/语言** | Python | 3.11+ | PaddleOCR、openpyxl、ReportLab 均为 Python 生态 |

| **API 协议** | RESTful JSON | — | 前后端分离的标准方案，直观易调试 |

| **OCR 引擎** | PaddleOCR | 2.8+ | 离线免费，中文识别优秀，支持 GPU（可选） |
| **PDF 生成** | ReportLab + PyPDF2 | 最新 | ReportLab 创建 A4 布局的 PDF，PyPDF2 合并多张发票 |
| **Excel 处理** | openpyxl | 3.1+ | 读写 .xlsx 格式，完全兼容台账模版格式 |
| **图片处理** | Pillow (PIL) | 10.x | 图片格式转换、缩放、裁剪，OCR 前预处理 |
| **文件存储** | 本地文件系统 | — | 个人使用，发票文件存储在本地的 `uploads/` 目录 |
| **环境变量** | python-dotenv | 最新 | 管理 `.env` 中的敏感配置 |

## 4. 数据存储 (Database)

| 维度 | 选型 | 说明 |
|------|------|------|
| **主数据库** | SQLite | 零配置、文件级存储、适合个人使用。数据库文件存储在 `data/` 目录下 |
| **缓存** | 不需要 | 个人使用场景，并发量极低，无需 Redis 缓存层 |
| **文件存储** | 本地文件系统 (`uploads/`) | 上传的发票 PDF/图片按日期和批次组织存储 |
| **未来迁移** | PostgreSQL | 如果需要多人协作或部署到服务器，SQLAlchemy ORM 可无缝切换 |

### 数据库核心表设计（概览）

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| `users` | 用户表 | id, username, password_hash, display_name |
| `invoices` | 发票表 | id, user_id, invoice_no, amount, date, category, vendor, file_path, status(pending/archived), ocr_raw_data |
| `reimbursement_batches` | 报销批次表 | id, user_id, department, period_start, period_end, reporter, reviewer, payee, bank_account, bank_name, total_amount, status |
| `batch_invoices` | 批次-发票关联表 | id, batch_id, invoice_id, expense_item, remark, is_substitute, substitute_for |
| `archives` | 归档记录表 | id, batch_id, archive_path, created_at |

## 5. 基础设施 & 工具 (Infra & Tools)

| 维度 | 选型 | 说明 |
|------|------|------|
| **部署方式** | 本地运行（开发&生产同一环境） | 个人桌面端工具，通过 `uvicorn main:app --host 0.0.0.0 --port 0220` 启动 |

| **包管理（后端）** | pip + requirements.txt 或 poetry | Python 标准包管理 |
| **版本控制** | Git | 代码管理 |
| **开发环境** | VS Code + Python / TypeScript 插件 | 推荐开发环境 |

## 7. 选型理由总结

| 需求点 | 选型如何满足 |
|--------|-------------|
| OCR 发票识别 | PaddleOCR（Python 生态）→ 决定后端必须 Python |
| 台账 Excel 导出 | openpyxl 读写 .xlsx，完全兼容台账模版格式 |
| 发票合并 PDF | ReportLab 创建 A4 布局，PyPDF2 合并多张发票图片/PDF |
| 费用报销单预览 | React 前端渲染 A4 纸风格页面，通过 @react-pdf-viewer 或 window.print() 打印 |
| Figma 设计稿还原 | Tailwind CSS 原子化样式 + Radix UI 无样式组件 = 自由定制，精确还原 |
| JWT 用户认证 | python-jose + passlib，轻量免审批流 |
| 替票管理 | ORM 层面通过 `batch_invoices` 关联表和 `is_substitute` 字段支持三种替票模式 |
| 归档管理 | `status` 字段标记 + 文件系统按批次目录分类存储 |
| 端口 0220 | Uvicorn 启动时指定 `--port 0220` |
