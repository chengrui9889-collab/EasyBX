# 项目开发路线图 (Development Roadmap)

## 0. 策略概览

- **开发策略**: 自底向上，先打地基（项目初始化 + 认证 + 数据库），再建核心业务（发票上传→OCR→批次→台账），最后补完善功能（替票→PDF→报销单预览）和优化增强。
- **MVP 目标**: 用户能从登陆到完成一次完整的报销闭环 —— 上传发票 → OCR 识别 → 人工校验 → 创建批次 → 勾选发票 → 导出 Excel 台账。整个流程跑通即为 MVP 完成。
- **依赖分析**: `Auth → Invoice → Batch → Excel台账` 是核心依赖链，每个上游模块完成后再开发下游。

## 2. 里程碑一：MVP — 核心报销闭环
> 目标：完成从 0 到 1 的报销全流程闭环。用户能登录 → 上传发票 → OCR 识别 → 创建报销批次 → 勾选发票 → 生成 Excel 台账。

### 2.1 用户认证 (Auth)
> 地基中的地基，所有功能都需要用户身份。

- [ ] **Phase 1.1: 认证 — 后端**
    - **模块**: `server/app/api/auth.py` + `server/app/services/auth_service.py`
    - **说明**: 用户注册（`POST /api/auth/register`）、登录（`POST /api/auth/login`）、JWT Token 生成与验证、`get_current_user` 依赖注入
    - **Pydantic Schema**: `RegisterRequest`、`LoginRequest`、`TokenResponse`
    - **依赖**: Phase 0.3（users 表）、Phase 0.2（数据库 Session）

- [ ] **Phase 1.2: 认证 — 前端**
    - **模块**: `web/src/pages/LoginPage.tsx` + `RegisterPage.tsx`
    - **说明**: 登录/注册页面 UI、Zustand authStore 管理 Token、axios 拦截器自动附加 `Authorization` 头、路由守卫（未登录跳转登录页）
    - **组件**: `web/src/components/ui/` 中的 Input/Button 等基础组件
    - **依赖**: Phase 0.4、Phase 1.1

### 2.2 发票管理 (Invoice CRUD + OCR)
> 核心数据入口，所有后续操作的数据来源。

- [ ] **Phase 1.3: 发票管理 — 后端**
    - **模块**: `server/app/api/invoices.py` + `server/app/services/invoice_service.py` + `server/app/services/ocr_service.py`
    - **说明**:
        - 发票上传（`POST /api/invoices`）→ 文件保存 → 触发 PaddleOCR 识别
        - OCR 服务：提取发票号、金额、日期、事由类别等字段
        - 发票列表查询（`GET /api/invoices`，支持 status/pending 筛选）
        - 单张发票查询（`GET /api/invoices/{id}`）
        - 发票信息更新/补全（`PUT /api/invoices/{id}`）
    - **Pydantic Schema**: `InvoiceResponse`、`UpdateInvoiceRequest`
    - **依赖**: Phase 1.1（需要用户认证）、Phase 0.3（invoices 表）

- [ ] **Phase 1.4: 发票管理 — 前端** ⭐ 重点 UI 交互
    - **模块**: `web/src/pages/InvoicesPage.tsx`
    - **说明**:
        - 发票上传区（拖拽/点击上传，显示上传进度）
        - OCR 识别结果展示卡片（发票号、金额、日期等）
        - 人工校验/编辑表单（React Hook Form）
        - 发票列表（分状态展示：待处理/已入库/已归档）
        - 发票筛选（按日期范围、按状态）
    - **组件**: `InvoiceUploader`、`InvoiceCard`、`InvoiceEditor`、`InvoiceList`、`InvoiceFilter`
    - **依赖**: Phase 1.3（后端 API）、Phase 0.4（基础组件）

### 2.3 报销批次 + 台账导出
> 核心业务逻辑，将发票组织为可报销的台账。

- [ ] **Phase 1.5: 报销批次管理 — 后端**
    - **模块**: `server/app/api/batches.py` + `server/app/services/batch_service.py`
    - **说明**:
        - 批次 CRUD（`GET/POST/PUT/DELETE /api/batches`）
        - 向批次添加/移除发票（`POST/DELETE /api/batches/{id}/invoices`）
        - 批次详情查询（含关联发票列表）
    - **Pydantic Schema**: `CreateBatchRequest`、`BatchResponse`、`BatchInvoiceRequest`
    - **依赖**: Phase 1.1（用户认证）、Phase 0.3（batches + batch_invoices 表）

- [ ] **Phase 1.6: 台账 Excel 导出 — 后端**
    - **模块**: `server/app/services/excel_service.py` + `server/app/api/exports.py`
    - **说明**:
        - 加载 `文件/台账模版.xlsx` 作为模版
        - 填入批次关联的发票数据（日期、事由、数量、单价、金额、垫款金额、备注）
        - 自动计算合计行
        - 填写表头（公司名、部门、报销期间）和底部（审核人、报账人、收款信息）
        - API: `GET /api/batches/{id}/export-excel` 返回 Excel 文件下载
    - **依赖**: Phase 1.5（批次含发票列表数据）

- [ ] **Phase 1.7: 报销批次 + 台账 — 前端**
    - **模块**: `web/src/pages/BatchesPage.tsx` + `BatchDetailPage.tsx`
    - **说明**:
        - 批次列表页：展示所有批次（状态、金额合计、日期）
        - 创建批次表单（部门、报销期间、报账人、收款信息）
        - 批次详情页：展示批次信息 + 关联发票列表
        - 发票选择器：从未报销发票库中勾选添加到当前批次
        - **台账导出台账下载按钮**：调用 `/api/batches/{id}/export-excel`
    - **组件**: `BatchForm`、`BatchCard`、`InvoiceSelector`、`LedgerPreview`
    - **依赖**: Phase 1.5、Phase 1.6（后端 API）、Phase 1.4（发票列表组件可复用）

### 2.4 概览页 (Dashboard)
> MVP 的收尾，提供入口和概览。

- [ ] **Phase 1.8: 概览页 — 前端**

## 3. 里程碑二：功能完善 (Feature Complete)
> 目标：补充 MVP 之外的核心功能 —— 替票管理、PDF 合并、费用报销单预览。

### 3.1 替票管理
> 真实报销场景的痛点功能，支持三种替票模式。

- [x] **Phase 2.1: 替票管理 — 后端**
    - **模块**: `server/app/services/batch_service.py`（扩展）
    - **说明**:
        - 批次内发票关联替票：`batch_invoices` 表的 `is_substitute` 和 `substitute_for` 字段逻辑
        - 替票规则支持：
            - **一对一替换**: 一张替票发票替换一笔费用 → `substitute_for` 指向被替换的费用行 ID
            - **一对多拆分**: 一张大额替票拆给多笔费用 → 同一张 `invoice_id` 对应多条 `batch_invoices` 记录
            - **多对一合并**: 多张小额替票合并覆盖一笔费用 → 多条 `batch_invoices` 记录的 `substitute_for` 指向同一费用 ID
        - 台账备注自动标注"替票"信息，格式如 `国际商务微课制作奖金（替票1011）`
    - **依赖**: Phase 1.5（批次管理基础）、Phase 0.3（batch_invoices 表结构）

- [x] **Phase 2.2: 替票管理 — 前端**
    - **模块**: `web/src/components/batches/SubstituteTicketModal.tsx`
    - **说明**:
        - 在批次详情页新增"设为替票"操作按钮
        - 替票关联弹窗：选择替票模式（一对一/一对多/多对一）、选择目标费用行
        - 替票标记展示：在台账预览中用特殊样式标注替票行
    - **依赖**: Phase 2.1（后端 API）、Phase 1.7（批次详情页 UI）

### 3.2 发票合并 PDF 导出
> 多张发票整理为可打印 PDF，支持 A4 纵向/横向排版。

- [ ] **Phase 2.3: PDF 导出 — 后端**
    - **模块**: `server/app/services/pdf_service.py` + `server/app/api/exports.py`（扩展）
    - **说明**:
        - ReportLab 创建 A4 页面布局
        - 支持两种排版模式：
            - **A4 纵向**: 一页放 2 张发票，上下排列
            - **A4 横向**: 一页放 4 张发票，2×2 排列
        - 发票图片/PDF 渲染到 A4 页面
        - 添加页码、页眉（公司名/批次号）等辅助信息
        - API: `POST /api/exports/invoice-pdf`
    - **依赖**: Phase 1.3（发票文件存储）

- [ ] **Phase 2.4: PDF 导出 — 前端**
    - **模块**: `web/src/components/exports/PdfExportSettings.tsx`
    - **说明**:
        - 在批次详情页或发票列表页增加"导出 PDF"按钮
        - 排版选择弹窗（纵向/横向）+ 发票勾选
        - 使用 @react-pdf-viewer 预览 PDF 导出效果
    - **依赖**: Phase 2.3（后端 API）、Phase 1.4（发票选择 UI）

### 3.3 费用报销单预览
> 按事由合并展示，A4 纸风格，可直接打印/导出 PDF。

- [ ] **Phase 2.5: 费用报销单 — 后端**
    - **模块**: `server/app/services/reimbursement_service.py` + `server/app/api/exports.py`（扩展）
    - **说明**:
        - 按批次关联发票按 `expense_item`（事由）分组求和
        - 生成费用报销单结构化数据（表头：部门/日期/单据数 + 表格行：事由/摘要/金额 + 底部合计）
        - API: `GET /api/exports/reimbursement-preview/{batch_id}`（JSON 预览数据）
        - API: `GET /api/exports/reimbursement-pdf/{batch_id}`（PDF 文件下载）
    - **依赖**: Phase 1.5（批次含发票数据）

- [ ] **Phase 2.6: 费用报销单预览 — 前端** ⭐ 核心 UI 交互
    - **模块**: `web/src/pages/ReimbursementPreviewPage.tsx` + `web/src/components/exports/ReimbursementForm.tsx`
    - **说明**:
        - 对应 Figma "报销单"设计稿，A4 纸风格页面
        - 左侧侧边栏 + 顶部工具栏 + 中间 A4 纸预览区域
        - 表格按事由合并展示（如 3 笔"打车费"合并为一行）
        - 顶部"打印/导出 PDF"按钮
        - **高优先级**：严格还原 Figma 设计稿中报销单的视觉细节（表头线框、表格边框、字体、签名区等）
    - **依赖**: Phase 2.5（后端 API）、Phase 1.2（认证）、Phase 0.4（基础组件）

## 8. 使用说明

1. **按顺序执行**：每个 Phase 都有标注依赖项，必须等所有依赖完成后才能开始。
2. **勾选进度**：每完成一个 Phase，将 `[ ]` 更新为 `[x]`。
3. **验收标准**：每个 Phase 完成后，启动前后端服务，手动验证 API 和 UI 是否正常工作。
4. **启动方式**：后端 `cd server && uvicorn main:app --host 0.0.0.0 --port 0220`，前端 `cd web && npm run dev`。
